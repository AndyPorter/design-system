Weeknotes

11 September 2020

I would like the way we work to be an exemplar of working in an agile way - be open, clear, communicate reguarly, not be too stiff, just enough, etc, so have:

  • started a kanban board for the next phase - a backlog of big things, that will feed the 'to-do now' list. Just me, Joe, Laura and Erin to start
  • added a history page. Good to show people that content is being added/improved (governance / demonstrate value), But more so to help design system users to find out 'whats new since i last came here?' The GOV design system - i find i don't know when advice gets updated, or new advice is added. I think that's a useful thing to provded a user. As this feels like it would be useful functionality - its something to test (note to self - add to trello cards for user need and + assumption/to test 😀)
  • started these 'project weeknotes' - good to a) reflect on what done/learnt and b) communicate progress in a simple, lightweight way
  • orgainsed a meet for next week to decide shape of next phase of work, and make a backlog

The advice published - the 'what do we currently have' - is being updated daily as I've been improving the collated documents, to ensure they are a single way to see what we have documented about each thing.

This housekeeping also includes the google sheet, so everything is tidy enough to be useful and less a source of confusion. This led me to think more about the purpose of the different things we have, and tweak things:

  • website - the front door - the place to go to for design advice and everything about the design system (and the project work - hence this page)
  • collated documents - where what we know about each component/pattern is collated. These are essentialy the 'pages' of what we currently do, that are published here. So needed to make each one 'the thing', by ensuring no orphan documents exist and that its all joined up. These documents are the public advice, and can have comments, dicscussion, opinion - they are what we know and think. Google docs are great to comment, track changes, collaborate etc - everyone can use with no technical training needed (unlike github). So an important principle/need (note to self - add to my list 😀😀) - minimimise the barrier to entry/collaboration
  • google sheet - use to track progress and see status - a view across the documentation - for example: whos working on documents, has it been reviewed. The sheet contains more than this, like proposed patterns / ideas for patterns - they would be better surfaced on the site, with a mechanism for feedback and new patterns spotted (😁 another note to self)

wow, writing this weeknote is useful - made me think, and i added 4 useful items to the backlog/what we know - user needs, principles, something to test/learn more about, how we orgainse the work so its easier to do 😝
